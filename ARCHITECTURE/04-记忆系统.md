# 04-记忆系统

> 本文档汇总 LifeButler 的双层记忆架构（五层文件版 + 七维 MySQL）
> 权威来源：`docs/memory-system-v2.md`
> 更新时间：2026-06-06

---

## 1. 两个系统并行

| 系统 | 回答的问题 | 核心用途 | 存储 |
|------|-----------|---------|------|
| **五层记忆** | 信息何时存在、如何衰减 | 追踪生命周期、时间维度 | Markdown 文件 |
| **七维记忆** | 信息是什么、有什么用 | 主动辅助决策、内容维度 | MySQL（`life_butler_db`）|

两个系统**完全独立**，不合并。

## 2. 五层记忆

### 2.1 文件结构

| 层级 | 文件路径 |
|------|---------|
| 瞬时 | `memory/transient.md` |
| 日记 | `memory/days/YYYY-MM-DD.md` |
| 周记 | `memory/weeks/YYYY-Www.md` |
| 月记 | `memory/months/YYYY-MM.md` |
| 年记 | `memory/years/YYYY.md` |

### 2.2 触发规则

#### 每次操作后 → 写 transient

```
文件：memory/transient.md
格式：什么操作 / 操作何处 / 操作前后状态
```

#### 每轮对话结束 → 写 days + 清空 transient

```
1. transient 内容 + 对话内容 → memory/days/YYYY-MM-DD.md
2. 清空 transient.md
```

#### 周期汇总

- 周一 → 汇总当周日记 → weeks/YYYY-Www.md
- 每月末/下月1日 → 汇总当月周记 → months/YYYY-MM.md
- 每年末/下年1日 → 汇总当年月记 → years/YYYY.md
- 年记写完后 → 提醒用户同步长期记忆到 MEMORY.md

### 2.3 日级记忆格式（核心原则）

> **原封不动**，不加工不摘要

```
[HH:MM:SS] <sender>: <原始消息>
[操作] 调用 xxx / xxx / 操作前：xxx / 操作后：xxx
[HH:MM:SS] <回复者>: <回复原文>
[操作] ... (transient 复制)
```

**关键点**：
- 消息原文一字不改
- 回复原文一字不改
- 操作记录只从 transient 复制

`skills/memory-layers-skill/`（含 SKILL.md + 5 个 TEMPLATES）

## 3. 七维记忆

### 3.1 七维维度

| 维度 | 记什么 | 主管 Agent |
|------|--------|-----------|
| **口味偏好** | 辣/清淡/海鲜/过敏源 | Trip Agent |
| **消费习惯** | 花钱频率/金额/场景 | Account Agent |
| **关系网络** | 联系人/关系/接触频率 | Schedule Agent |
| **时间规律** | 作息/工作节奏/空闲时段 | Schedule Agent |
| **遗忘清单** | 说过两次想吃还没去 | Schedule Agent |
| **认知风格** | 详细 vs 简洁、逻辑 vs 感觉 | Coordinator |
| **健康情况** | 睡眠/运动/过敏/疾病 | Coordinator |

### 3.2 核心原则

- 任何 agent 都可以往任何维度的 cache 写侧写
- 主管 agent 的职责是降级管理（不是写权限控制）
- 真冲突：同维度、矛盾结论（才需要处理）
- 不同 agent 写的不同侧面不算冲突
- 证据链要求：每条总结性记录必须可追溯到原始对话

### 3.3 MySQL 表结构

#### cache_events（第一层：未验证侧写）

```sql
CREATE TABLE cache_events (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    dimension   VARCHAR(32)      NOT NULL,
    sub_key     VARCHAR(64)      NOT NULL,
    content     TEXT             NOT NULL,
    evidence    TEXT             NOT NULL,
    evidence_list JSON           NOT NULL,
    agent_id    VARCHAR(64)      NOT NULL,
    source_ref  VARCHAR(128)     DEFAULT '',
    weight      TINYINT          DEFAULT 1,
    demoted     TINYINT          DEFAULT 0,
    created_at  DATETIME         NOT NULL,
    expires_at  DATETIME         NOT NULL,
    INDEX idx_dimension_created (dimension, created_at),
    INDEX idx_expires (expires_at)
);
```

#### seven_dimensions（第二层：已验证画像）

```sql
CREATE TABLE seven_dimensions (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    dimension       VARCHAR(32)      NOT NULL,
    sub_key         VARCHAR(64)      NOT NULL,
    content         TEXT             NOT NULL,
    evidence        TEXT             NOT NULL,
    evidence_list   JSON             NOT NULL,
    agent_id        VARCHAR(64)      NOT NULL,
    source_ref     VARCHAR(128)     DEFAULT '',
    confidence      TINYINT          NOT NULL,
    status          VARCHAR(16)      DEFAULT 'active',
    demoted_at      DATETIME         DEFAULT NULL,
    promoted_at     DATETIME         NOT NULL,
    updated_at      DATETIME         NOT NULL,
    UNIQUE KEY uk_dimension_subkey (dimension, sub_key),
    INDEX idx_status (status)
);
```

#### promote_log（晋升日志）

```sql
CREATE TABLE promote_log (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    cache_event_ids JSON             NOT NULL,
    target_id       BIGINT           NOT NULL,
    target_type     VARCHAR(16)      NOT NULL,
    created_at      DATETIME         NOT NULL
);
```

#### emergency_events（突发事件临时覆盖层）

```sql
CREATE TABLE emergency_events (
    id                  BIGINT AUTO_INCREMENT PRIMARY KEY,
    event_id            VARCHAR(64)      NOT NULL UNIQUE,
    dimension           VARCHAR(32)      NOT NULL,
    override_key        VARCHAR(64)      NOT NULL,
    override_value      TEXT             NOT NULL,
    source              VARCHAR(32)      DEFAULT NULL,
    start_date          DATE             NOT NULL,
    end_date            DATE             NOT NULL,
    duration_days       SMALLINT         NOT NULL,
    check_interval_days SMALLINT         DEFAULT 2,
    next_check_date     DATE             NOT NULL,
    status              VARCHAR(16)      DEFAULT 'active',
    resume_profile      BOOLEAN          DEFAULT TRUE,
    created_at          DATETIME         NOT NULL,
    updated_at          DATETIME         NOT NULL,
    INDEX idx_next_check (next_check_date, status)
);
```

## 4. 七条路径

### 路径1：写入侧写

**写入顺序（关键）**：
```
① 先写五层 days/YYYY-MM-DD.md → 得到路径+行号
② 再写 cache_events（source_ref = "memory/days/2026-06-03.md-第45行-摘要"）
```

**四种合并情况**：

| cache | dimension | 操作 |
|-------|-----------|------|
| 无 | 无 | cache_events 新增，weight=1 |
| 有 | 无 | 追加 evidence_list，weight+1 |
| 无 | 有 | 追加 evidence_list，confidence+1 |
| 有 | 有 | 不可能（promote 后 cache 已清）|

### 路径2：Promote 晋升（双路线）

| 路线 | 触发 | 执行者 |
|------|------|--------|
| A 定时脚本 | 每天凌晨 4:00（兜底）| cron（🔄 可选，保底用）|
| B 主管Agent顺手 | 路径1执行中实时发现 weight≥3 | 主管 Agent |

**条件**：同一 dimension+sub_key，14 天内 ≥3 条 → 晋升。

### 路径3：查询（Coordinator 决策前）

**优先级**：emergency_events > seven_dimensions > cache_events

### 路径4：降级（仅主管 Agent）

**触发**：用户行为与 dimension 侧写矛盾。

**4 步流程**：
```
1. 写矛盾侧写到 cache
2. Coordinator 询问用户
3. 综合判断
4. 主管 Agent 降级：cache 写"降级侧写"标注（1年观察期），dimension status='demoted'
```

### 路径5：突发事件（Coordinator 主管）

**来源**：
- 用户直接提到（"最近口腔溃疡"等）
- 路径4触发后用户解释

**写入 emergency_events**，有 duration_days 和 next_check_date。

**Coordinator 路由前必查**。

### 路径6：遗忘清单主动服务

**特点**：
- 不过 cache，直接写入 seven_dimensions
- 由 Coordinator / Schedule Agent 偶尔查看
- 联合时间规律判断是否提醒

### 路径7：聊天模式

**意图不明显 → 聊天模式**：
- 只写五层（transient → days）
- 不写七维 cache
- 聊天结束后扫拈识别侧写（由 **subagent** 完成，不用脚本调 LLM）

**Subagent 设计**：
- 用 OpenClaw `sessions_spawn` 工具
- Subagent 本身是 LLM，不绕弯（脚本调 LLM = 一个 LLM 委托另一个 LLM）
- 后台任务可用便宜模型（如 MiniMax-M2.7）
- 提示词位置：`skills/memory-seven-dim-skill/SKILL.md` 路径7
- 调用规范见 `skills/subagent-skill/SKILL.md`（含 sessions_spawn 参数）
- Cron 触发见 `ARCHITECTURE/06-Cron-定时任务.md` 任务 9

## 5. 已定参数

| 参数 | 值 | 依据 |
|------|-----|------|
| 触发阈值 | **3 次** | 薄片研究：3 次一致性行为足以形成稳定特质 |
| 时间窗口 | **14 天** | 覆盖 2 个完整工作周期+周末 |
| 降级触发 | 1 次明确矛盾 → 候选降级 | 确认偏误：推翻比建立难 |
| 降级后 cache 观察期 | **1 年** | 降级侧写需要更长观察期 |
| 多 agent 写入 | 任何 agent 可写任何维度 | — |
| 真冲突 | 同维度矛盾结论 | — |
| 遗忘清单 | 不过 cache，直接写 dimension | 多半不重复 |

## 6.5 首次使用流程（init-readme）

**背景**：库初始化时 `seven_dimensions` 是空的。改为让用户填问卷。

**触发**：Coordinator 检测到 `seven_dimensions` 表为空 → 读 `coordinator/memory/init-readme.md`。

**问卷**（Q1-Q13，分为两模块）：

基础模块 Q1-Q7（认知风格/健康/偏好/预算/地点/时间/社交）：
- Q1 回答风格 → 认知风格/输出详细度
- Q2 运动习惯 → 健康情况/运动频率
- Q3 旅行偏好 → 旅行偏好/活动类型
- Q4 预算档次 → 消费习惯/预算档次
- Q5 出行城市 → 旅行偏好/常驻城市
- Q6 可用时间 → 时间习惯/工作节奏
- Q7 出行人数 → 社交偏好/出行人数

财务模块 Q8-Q13（可选跳过）：
- Q8 月收入段 → 财务状况/收入段
- Q9 收入来源 → 财务状况/收入来源
- Q10 必要支出 → 财务状况/必要支出
- Q11 存钱目标 → 财务状况/存钱意图
- Q12 存钱目标详情 → 财务状况/存钱目标（effort_goals）
- Q13 负债情况 → 财务状况/负债情况

**处理**：
- Coordinator 直接渲染 skill 中的问卷内容，发送用户
- 用户提交后：answers JSON 写入 `coordinator/data/init_questionnaire.json`
- Coordinator 读取 JSON → 生成 MySQL INSERT 写入 `seven_dimensions`（agent_id='init'，confidence=3）
- Q12 选了 A（有存钱目标）→ 额外写入 `effort_goals.goals[]` 到 Account 的 accounts.json
- 完成后：标记 `init_completed: true` 到 `coordinator/memory/seven_dim_cache.json`，删除 `coordinator/memory/init-readme.md`

**与其他路径兼容**：
- 不经 cache_events，直接入 seven_dimensions（问卷可信度高）
- 可被后续 cache / emergency 覆盖

**详见**：`coordinator/memory/init-readme.md`
