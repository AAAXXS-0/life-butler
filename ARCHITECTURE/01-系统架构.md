# 01-系统架构

> 本文档描述 LifeButler 的总体架构、组件关系、数据流
> 更新时间：2026-06-06

---

## 1. 核心定位

> **"你只管生活，琐事交给我。"**
> LifeButler 不是帮你"找"东西，而是替你"安排"生活。

## 3. 组件清单

### 3.1 Agents（4 个）

| Agent | 文件位置 | 职责 |
|-------|---------|------|
| **Coordinator** | `coordinator/` | 意图识别、路由、组装回复、主动服务 |
| **Trip Agent** | `agents/trip-agent/` | 行程规划、动态调整、空间优化 |
| **Schedule Agent** | `agents/schedule-agent/` | 日程备忘、时间提醒、习惯追踪 |
| **Account Agent** | `agents/account-agent/` | 账本管理、消费分析、预算预警 |

### 3.2 Skills（8 个）

| Skill | 文件位置 | 谁用 | 作用 |
|-------|---------|------|------|
| `trip-skill` | `skills/trip-skill/` | Trip Agent | 4 阶段行程规划 |
| `butler-comm-skill` | `skills/butler-comm-skill/` | 所有 Agent | 跨 Agent 异步通信 |
| `memory-layers-skill` | `skills/memory-layers-skill/` | 所有 Agent | 五层记忆写入 |
| `memory-seven-dim-skill` | `skills/memory-seven-dim-skill/` | 所有 Agent | 七维记忆读写 |
| `poi-skill` | `skills/poi-skill/` | Coordinator | 通用地点查询 |
| `meal-skill` | `skills/meal-skill/` | Coordinator | 餐厅/外卖查询 |
| `fun-skill` | `skills/fun-skill/` | Coordinator | 演出/展览/活动 |
| `replan-skill` | `skills/replan-skill/` | Coordinator | 随机事件处理 |
| `subagent-skill` | `skills/subagent-skill/` | 所有 Agent | sessions_spawn 工具使用规范 |

### 3.3 共享数据（`data/`）

| 文件 | 核心写入者 | 其他 Agent 读 |
|------|----------|--------------|
| `schedule.json` | Schedule Agent + Trip Agent（写 event 碎片） | Trip、Account 读 |
| `accounts.json` | Account Agent | Trip、Schedule 读 |
| `trips.json` | Trip Agent | Schedule、Account 读 |

> **业务数据库管理原则**：写操作一对一（专属 Agent），读操作共享。

### 3.4 POI 数据（`poi/`，图模型迁移中）

| 文件 | 内容 | 状态 |
|------|------|------|
| `attractions.json` | 景点数据 | 图模型后改为 MySQL |
| `restaurants.json` | 餐厅数据 | 图模型后改为 MySQL |
| `hotels.json` | 酒店数据 | 图模型后改为 MySQL |
| `intercity.json` | 城际交通 | 图模型后改为 MySQL |

### 3.5 通信通道（`shared/`）

```
workspace-butler/shared/
├── coordinator-to-trip-agent/<日期>.json
├── trip-agent-to-coordinator/<日期>.json
├── coordinator-to-schedule-agent/<日期>.json
├── ...（共 12 个通道，2 个备用）
└── taxonomy.json
```

详细见 [03-通信协议.md](./03-通信协议.md)

## 4. 数据流

### 4.1 用户消息 → 响应

```
用户消息
    ↓
Coordinator 收到
    ↓
查 emergency_events（突发事件）
    ↓ 有 → 处理突发事件
    ↓ 无 → 继续
    ↓
意图识别（语义分析 + 七维画像辅助）
    ↓
路由：
  ├─ 找吃的/玩的/地点 → 直接调 skill
  ├─ 出游/记日程/记账 → butler-comm 委托 Agent
  └─ 多 Agent 协作 → Gather Session 聚合
    ↓
组装回复 → 用户
```

### 4.2 Agent 委托 → 任务完成

```
Coordinator
    ↓ 写 task_delegate 到 shared/<coordinator> to <agent>/
    ↓ openclaw cron run <agent-job-id>
    ↓
目标 Agent 醒来
    ↓ 读 shared/ 消息
    ↓ 读 data/ 相关文件（直接读，不通信）
    ↓ 处理业务
    ↓ 写 result_callback 到 shared/<agent> to coordinator/
    ↓ openclaw cron run coordinator-job-id
    ↓
Coordinator 醒来
    ↓ 检查 Gather Session（如有）
    ↓ 读 shared/ 回复
    ↓ 组装给用户
```

## 5. 核心原则

1. **数据共享优先**：所有 Agent 共享 `data/` 目录，读操作无限制
2. **直接读 > 通信**：能直接读文件就不走通信
3. **写操作专属**：每个数据文件只有一个 Agent 主导写入
4. **异步优先**：Agent 之间用 butler-comm（async + 文件 + cron）
5. **语义分析**：Coordinator 用 LLM 理解意图，不用关键词匹配

## 6. 状态对比（v1 → v2 → v3）

| 维度 | v1 (2026-05) | v2 (2026-05-25) | **v3 (2026-06-05)** |
|------|--------------|------------------|---------------------|
| 数据 | 静态 JSON | 静态 JSON + 通信 | 图模型（MySQL） |
| Agent 通信 | 各自独立 | sessions_send 互相通信 | 共享 data + 但ler-comm 异步 |
| POI 路径 | `agents/.../data/` | `agents/.../data/` | `poi/` 顶层（待图模型落地后入 MySQL） |
| 触发方式 | sessions_send | sessions_send | cron run（不用 sessions_send） |
| 事件处理 | 手动 | 手动 | 7 种事件自动生成 + replan |
| 记忆 | 散落 | 散落 | 五层文件版 + 七维 MySQL 并行 |

**v3 是当前生效版本**。
