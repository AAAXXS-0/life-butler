# 03-通信协议

> 本文档描述 Agent 之间、Agent 与 Coordinator 之间的通信机制
> 权威来源：`skills/butler-comm-skill/SKILL.md`
> 更新时间：2026-06-06

---

## 1. 核心机制

**触发方式**：**openclaw cron run `<job-id>`**（不用 sessions_send）

**消息载体**：`shared/<sender> to <receiver>/YYYY-MM-DD.json`（按天分割的 JSON 文件）

**为什么用 cron run**：
- ✅ 稳定可靠（sessions_send 经常出问题）
- ✅ 异步处理（agent 离线也能被唤醒）
- ✅ 可链式触发

---

## 2. 目录结构

**核心原则**：cron job 有方向性（`openclaw cron run A` 是单向触发），但聊天文件夹是双向的（任何人都可以往里写、读走别人的消息）。

**因此：每个 agent 一个收件箱文件夹**，方向靠消息里的 `from` 字段区分。

```
workspace-butler/shared/
├── coordinator/           ← coordinator 的收件箱（任何 agent/0 可写）
├── trip-agent/            ← trip-agent 的收件箱（任何 agent 可写）
├── schedule-agent/        ← schedule-agent 的收件箱（任何 agent 可写）
├── account-agent/         ← account-agent 的收件箱（任何 agent 可写）
└── taxonomy.json          ← 通讯录（静态文件）
```

**4 个收件箱**（精简掉双向分离的 12 通道）。**从 12 路径精到 4 路径**。

**示例**：
- coordinator 要委托 trip → 写 `shared/trip-agent/YYYY-MM-DD.json`（今天日期），调 `openclaw cron run <trip-wake-job-id>`
- trip 要回复 coordinator → 写 `shared/coordinator/YYYY-MM-DD.json`（今天日期），调 `openclaw cron run <coordinator-wake-job-id>`
- account 要推 trip（中间状态）→ 写 `shared/trip-agent/YYYY-MM-DD.json`（今天日期），调 `openclaw cron run <trip-wake-job-id>`

每条消息的 `from` 字段标识真实发送方，读取时根据 `from` 过滤（例：trip-agent 跑起来后只处理 `from != "trip-agent"` 的未读消息）。

---

## 3. 消息格式

**每日一个 JSON 文件**：

```json
[
  {
    "id": "<uuid>",
    "from": "<agent name>",
    "reply_to": "<uuid or null>",
    "ts": 1749087613000,
    "content": "<正文内容>",
    "read": false,
    "type": "task_delegate|result_callback|info_share"
  }
]
```

**字段**：
| 字段 | 说明 |
|------|------|
| `id` | 消息唯一 ID |
| `from` | 发送方 agent 名称 |
| `reply_to` | 回复的消息 ID |
| `ts` | 时间戳（毫秒） |
| `content` | 正文 |
| `read` | 已读标记 |
| `type` | `task_delegate` / `result_callback` / `info_share` |

---

## 4. 委托 Agent 完整流程

```
1. Coordinator 写 task_delegate 到 shared/<coordinator> to <agent>/YYYY-MM-DD.json
2. Coordinator: openclaw cron run <agent-job-id>
3. 目标 Agent 醒来
4. Agent 读 shared/ 消息 → 标记 read=true
5. Agent 处理业务（读 data/、调 skill）
6. Agent 写 result_callback 到 shared/<agent> to coordinator/YYYY-MM-DD.json
7. Agent: openclaw cron run coordinator-job-id
8. Coordinator 醒来
9. Coordinator 读 shared/ 回复 → 组装给用户
```

**禁止使用 sessions_send 触发 Agent**（不稳定）。

## 5. Gather Session（多 Agent 回复聚合）

**场景**：Coordinator 同时委托多个 Agent，必须等全部回复后统一回复用户。

### 5.1 文件结构

**位置**：`coordinator/memory/gather/<gather_id>.json`

```json
{
  "gather_id": "gather_<uuid>",
  "task": "规划北京3日游，需查预算",
  "created_at": 1749087613000,
  "contacted": ["trip-agent", "account-agent"],
  "expected_count": 2,
  "replies": {
    "trip-agent": { "replied": false, "replied_at": null },
    "account-agent": { "replied": false, "replied_at": null }
  },
  "status": "pending|completed|timeout"
}
```

**关键设计**：
- ✅ `replies` 不存 content（内容在 shared/ 那边）
- ✅ gather 只管"谁回复了"，不管"回复了什么"
- ✅ Coordinator 组装回复时去 shared/ 读内容

### 5.2 流程

**发起**（Coordinator → 多 Agent）：
```
1. 写 task_delegate 到每个 Agent 的 shared/ 通道
2. 创建 gather session 文件
3. cron run 每个目标 Agent
```

**回复**（Agent → Coordinator）：
```
1. Agent 写 result_callback 到 shared/<agent> to coordinator/
2. 更新 gather session 中该 agent 的 replied=true
3. cron run coordinator
```

**检查**（Coordinator 醒来时）：
```
- 有 pending gather？
  - 全部 replied → 去 shared/ 读各 Agent 回复，组装给用户
  - 部分 replied → 继续等待，不回复用户
  - 超时 5 分钟 → 用已有回复组装，缺失标记"待确认"
```

## 6. 直接读文件原则（替代通信）

**核心原则**：数据文件本来就是共享的，能直接读就不走通信。

### 6.1 适用场景

| 场景 | 方式 |
|------|------|
| Trip 查时间冲突 | **直接读** `data/schedule.json` |
| Trip 查预算 | **直接读** `data/accounts.json` |
| Account 查关联 | **直接读** `data/schedule.json` |
| Schedule 查开销 | **直接读** `data/accounts.json` |
| Trip 写 event 碎片 | **直接写** `data/schedule.json`（source: "trip"） |

### 6.2 必须通信的场景

只有"对方要做处理"时才通信：

| 场景 | 理由 |
|------|------|
| Trip → Account（行程结束汇总） | Account 要做汇总 |
| Schedule → Account（宴请关联账本） | Account 要做关联 |
| Account → Schedule（开销关联日程） | Schedule 要做关联 |

## 9. 通讯用 Cronjob 通用参数配置

**所有 A2A 通讯都走 cron job**。本节定义**通用参数**（所有 A2A 任务都需遵守）。

### 9.1 必带参数

| 参数 | 取值 | 理由 |
|------|------|------|
| `--no-deliver` | 固定 | 不外发到聊天（内部传递） |
| `--expect-final` | 固定 | 等待 agent 处理完拿到结果（链式同步） |
| `--session` | `session:butler-<agent>:inbox` | **per-session 串行排队**，防 trip 并行跑 / 文件锁撞 / agent 被打断 |

**--session 是关键**：默认 `isolated` 每次新 session → 同一目标 agent 多 session 并行 → 写 shared/ 撞锁 + 上下文丢失。**必须**用 session-keyed inbox 串行排队。

### 9.2 命名约定

| 名称 | 模式 |
|------|------|
| 醒型（被 cron run 触发的"信使"） | `butler-<agent>-wake` |
| 主动服务型 | `butler-<agent>-proactive` |
| 整点巡检型 | `butler-coordinator-hourly-sweep` |

### 9.3 启用状态

| 类型 | `enabled` | 理由 |
|------|----------|------|
| 醒型（所有 wake） | `false`（死） | 只能被上游 `cron run` 触发，**绝不自跑** |
| 整点巡检型 | `true` | 定时任务，需自动跑 |
| 主动服务型 | `false` | 由整点巡检 `cron run` 链式触发 |

**⚠️ 警告**：**禁止**用 `openclaw cron enable <wake-job-id>`（会变成全局定时心跳，打破 A2A 触发链）。

### 9.4 通用注册命令

```bash
# A2A 醒型 job 通用模板
openclaw cron add \
  --name "butler-<agent>-wake" \
  --agent "butler-<agent>" \
  --session "session:butler-<agent>:inbox" \
  --message "<读 shared/<agent>/YYYY-MM-DD.json + 处理 + 写回复 + 触发下一跳>" \
  --no-deliver \
  --expect-final \
  --disabled
```

### 9.5 清单

实际注册的 A2A cron job（详见 `update_docs/butler-cronjobs-spec.md`）：

| name | enabled | 触发者 |
|------|---------|--------|
| `butler-trip-agent-wake` | false | coordinator / 其它 agent |
| `butler-schedule-agent-wake` | false | coordinator / 其它 agent |
| `butler-account-agent-wake` | false | coordinator / 其它 agent |
| `butler-coordinator-wake` | false | 子 agent 跑完后 |
| `butler-self-wake`（备用） | false | 0（主助手） |
| `butler-coordinator-hourly-sweep` | **true** | 定时（每整点） |
